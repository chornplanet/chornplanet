# Feature Plan: LLM Visibility Files

Branch for planning: `feature/mcp-agent-workspace`
Implementation branch: `feature/llm-visibility-files`
Status: Planning
Owner: Khachornchit Chief Architect → ChatGPT planning → Codex review/implementation

## Problem Statement

ChornPlanet needs public AI-readable files that help search engines and AI assistants understand the platform direction, major content categories, commerce direction, Chorn DNA / civilization identity, Smart Food evolution, and public navigation priorities.

The requested assets are:

```text
llm.txt
llm-full.txt
```

These should improve ChornPlanet's AI discoverability without exposing private repository details, secrets, or internal strategy that should not be public.

## Goals

- Add public LLM visibility files for ChornPlanet.
- Provide a concise public platform summary in `llm.txt`.
- Provide a richer public platform guide in `llm-full.txt`.
- Align with `.mcp/workflows/seo-llm-visibility.md` and `.mcp/tools/seo-llm-tools.yaml`.
- Keep the files accurate, public-safe, and useful for AI assistants.
- Support future growth toward 10K, 100K, 500K, and 1M daily traffic.

## Non-Goals

- Do not expose private repo details.
- Do not expose secrets, credentials, internal marketplace credentials, or analytics IDs.
- Do not claim live commerce integrations that are not yet live.
- Do not implement a full SEO system in this feature.
- Do not modify production analytics configuration.

## Existing Architecture Review

ChornPlanet already has SEO metadata infrastructure under `src/metadata/`, public routes, sitemap logic, and Next.js metadata routes such as robots/sitemap.

This feature should fit into the existing Next.js routing and public file strategy.

Potential implementation choices:

Option A: public static files

```text
public/llm.txt
public/llm-full.txt
```

Option B: app route handlers

```text
src/app/llm.txt/route.ts
src/app/llm-full.txt/route.ts
```

Codex should inspect the current app and choose the lowest-risk implementation that works with the deployed Next.js/static output setup.

## Proposed Content Structure

### `llm.txt`

Concise sections:

- ChornPlanet overview
- Main public categories
- Media platform direction
- Commerce direction
- ChornPlanet civilization / outfit direction
- Smart Food direction
- Public navigation hints

### `llm-full.txt`

Richer sections:

- Platform overview
- Audience and use cases
- Content categories
- News / entertainment / technology follow-up
- Chorn DNA / outfit / civilization posting direction
- Premium commerce and marketplace direction
- Smart Food evolution direction
- Luxury project direction
- Analytics and growth direction
- Public page discovery hints
- Contact / official site reference if available

## Project Structure Guideline

Preferred simple structure:

```text
public/llm.txt
public/llm-full.txt
```

If static files are not appropriate, use route handlers:

```text
src/app/llm.txt/route.ts
src/app/llm-full.txt/route.ts
```

Do not place these files under `.mcp` only, because the goal is public AI discoverability.

## Testing Plan

- Run lint if route handlers or code are added:

```bash
yarn lint
```

- Run build before merge:

```bash
yarn build
```

- Verify URLs:

```text
/llm.txt
/llm-full.txt
```

- Confirm files do not expose secrets or internal-only details.
- Confirm public claims are accurate and future-facing wording is clear.

## Risks and Open Questions

- Should the files be static or generated by route handlers?
- Should `llms.txt` also be supported as an alias later?
- Which public pages should be listed first?
- Should commerce sections say “planned” until marketplace integrations are live?

## Acceptance Criteria

- `llm.txt` is publicly accessible.
- `llm-full.txt` is publicly accessible.
- Content is public-safe and aligned with ChornPlanet platform direction.
- No secrets or private repo details are exposed.
- Build/lint checks pass if code/routes are added.
- `.mcp/workflows/seo-llm-visibility.md` remains the governing workflow for future updates.
