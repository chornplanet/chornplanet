# Chorn Planet Platform Roadmap

## Purpose

**Chorn Planet Platform** is a strategic AI-native business platform for managing and scaling Chorn Planet’s portfolio, initiatives, AI adoption, value capture, governance, technical coordination, and execution engine.

It is not only a content project, media brand, restaurant automation project, or AI experiment.

Chorn Planet Platform is a business platform that connects:

- AI media
- Smart food delivery
- LINE Bot ordering
- MenuMatch
- Bill processing
- AI dashboards
- AI document search
- Value tracking
- Portfolio governance
- Executive decision-making
- Laravel-based execution engine

The platform should support current and future Chorn Planet business initiatives.

## 5.1 ROI/SLA Methodology

Each initiative should define:

| Area               | Description                                         |
| ------------------ | --------------------------------------------------- |
| Business Objective | What business result the initiative should create   |
| ROI Metric         | How value will be measured                          |
| SLA/SLO/SLI        | How reliability or service quality will be measured |
| Baseline           | Current state before implementation                 |
| Target             | Expected result after implementation                |
| Measurement Period | Daily / Weekly / Monthly / Quarterly                |
| Owner              | Person responsible for value realization            |

Example:

```text
Initiative:
Bill Processing

Business Objective:
Reduce manual payment verification effort.

Baseline:
Finance manually checks payment slips.

Target:
AI-assisted bill processing verifies payment details faster and reduces manual workload.

ROI:
Time saved per order, reduced manual checks, fewer payment confirmation delays.

SLA:
Correct payment verification and response within target time.

SLI:
Verification success rate, error rate, average verification time.
```

---

## 5.2 Pre/Post Measurement

Every initiative should compare before and after results.

Example measurement model:

| Measurement              |          Before |          After |         Benefit |
| ------------------------ | --------------: | -------------: | --------------: |
| Manual verification time | 5 minutes/order | 1 minute/order |   80% reduction |
| Customer response time   |      10 minutes |       1 minute | 90% improvement |
| Order matching error     |            High |            Low | Better accuracy |
| Content production time  |           1 day |         1 hour |   Faster output |

---

## 10.2 Queue

Queue should be used for asynchronous work that does not need to block the customer immediately.

Good queue candidates:

```text
Send notification
Generate executive summary
Process uploaded document
Analyze dashboard metrics
Sync external data
Generate report
Run background AI task
Retry failed integration
```

Queue should not be used blindly for every action.

Synchronous handling is better when the user needs immediate confirmation, such as:

```text
Create order
Confirm current stock
Validate required input
Return menu price
Confirm payment result when immediate response is required
```

---

## 10.3 Redis

Redis should support:

```text

```
